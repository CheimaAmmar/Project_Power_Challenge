
# Power Side-Channel Attack on a Remote AES-128 Embedded Device

## Abstract

This repository presents the analysis and resolution of a hardware security challenge based on a **power side-channel attack** against a remote AES-128 embedded device. The purpose of the challenge is not to break AES from a mathematical perspective, but rather to exploit the **physical leakage** generated by its implementation during execution.

By collecting power consumption traces associated with chosen plaintext encryptions and applying **Correlation Power Analysis (CPA)** with a Hamming Weight leakage model, it becomes possible to recover the full 128-bit secret key. This project illustrates a fundamental concept in embedded security: even when a cryptographic algorithm is theoretically secure, its physical implementation may still leak enough information to compromise confidentiality.

---

## 1. Introduction

Cryptographic algorithms such as AES are designed to provide strong security guarantees. However, these guarantees usually assume an idealized implementation model. In practice, embedded systems execute cryptographic operations on physical hardware, and this hardware may unintentionally reveal information through side channels such as:

- power consumption,
- timing variations,
- electromagnetic radiation,
- acoustic emissions.

A **side-channel attack** exploits such physical leakage rather than a flaw in the algorithm itself.

In this challenge, the attacker is given remote access to an embedded device performing AES-128 encryption. For each chosen plaintext, the device returns a corresponding **power trace**, representing the measured power consumption during execution. The objective is to analyze these traces and recover the secret encryption key.

5. The attacker analyzes the traces to recover the secret key.

This corresponds to a **chosen-plaintext power analysis scenario**, which is a realistic threat model in hardware security and embedded cryptography.

## 3. From Algorithmic Security to Physical Leakage

AES remains secure as a cryptographic primitive. The attack does **not** exploit a weakness in AES itself. Instead, it targets the way AES is executed on physical hardware.

In digital circuits, and especially in CMOS technology, power consumption varies depending on the switching activity of internal transistors. As a result, intermediate values manipulated during encryption can influence the measured power signal.

This means that a secret-dependent internal state may leak through physical behavior, even though it is never explicitly output by the device.

This is the central idea behind power analysis attacks:

> the implementation leaks information that the algorithm does not.

---

## 4. Leakage Model

To exploit the measured traces, a leakage model must be defined.

A common and effective assumption is the **Hamming Weight model**, which states that the power consumption is correlated with the number of bits set to `1` in the manipulated intermediate value.

For an 8-bit value:

- `0x00 = 00000000` → Hamming Weight = 0
- `0x0F = 00001111` → Hamming Weight = 4
- `0xFF = 11111111` → Hamming Weight = 8

Although real hardware leakage is often more complex, the Hamming Weight model is sufficiently accurate in many educational and practical attacks.

---

## 5. Why the First AES Round Is Targeted

The attack focuses on the first round of AES because the corresponding intermediate values depend on:

- the **known plaintext**
- the **unknown key**

For each plaintext byte (P) and key byte (K), AES computes:

S = SBOX(P ⊕ K)

Since the plaintext is known, we can test every possible key guess (k ∈ [0,255]) and compute:

S_k = SBOX(P ⊕ k)

The predicted leakage is:

L_k = HW(S_k)

This provides a candidate power-consumption pattern for each possible key byte. The correct key hypothesis should produce a pattern that correlates more strongly with the real measured traces than incorrect hypotheses.

---

## 6. Attack Principle: Correlation Power Analysis

The key-recovery method used in this project is **Correlation Power Analysis (CPA)**.

CPA is a statistical attack that compares:

- a **predicted leakage** derived from a key hypothesis,
- the **actual measured power traces**.

The attack proceeds as follows:

### 6.1 Trace Acquisition

A large number of random plaintexts are sent to the target device. For each plaintext, the corresponding power trace is recorded.

This results in a dataset of:

- chosen plaintexts,
- measured traces.

### 6.2 Hypothesis Generation

For each AES key byte, the attacker tests all 256 possible values.

For each hypothesis, the predicted leakage is computed using:

HW(SBOX(P ⊕ k))

### 6.3 Correlation Computation

The predicted leakage vector is compared against the measured traces using the **Pearson correlation coefficient**.

For each key hypothesis, the maximum correlation across all time samples is retained.

The key candidate with the highest correlation is selected as the most likely correct byte.

### 6.4 Full Key Reconstruction

The CPA is repeated independently for all 16 bytes of the AES-128 key. The recovered bytes are then concatenated to reconstruct the full secret key.

### 6.5 Verification

The recovered key is submitted to the challenge service through the verification interface. If the key is correct, the service returns the flag.

## 7. Role of the Provided `socket_interface.py`

The file provided with the challenge is a **communication helper**, not the actual solver.

It allows the user to:

- connect to the remote service,
- send an option to the menu,
- send a plaintext,
- receive a base64-encoded trace,
- decode the trace into a NumPy array,
- submit a candidate AES key for verification.

In other words, it provides the **transport layer** required to interact with the device, but it does not include the attack logic itself.

The following core elements are intentionally absent from the provided script:

- leakage modeling,
- AES S-box processing,
- Hamming Weight computation,
- correlation analysis,
- key-byte recovery,
- full key reconstruction.

These components constitute the actual side-channel attack and must be implemented separately.

## 8. Solver Overview

The custom solver extends the provided interface into a complete attack pipeline. Its main tasks are:

- collecting multiple traces using random plaintexts,
- decoding and storing the traces,
- filtering malformed or inconsistent acquisitions,
- normalizing the traces,
- computing the CPA score for each key hypothesis,
- recovering each key byte independently,
- reconstructing and verifying the full AES key.

The core leakage expression used in the implementation is:

```python
HW[AES_SBOX[np.bitwise_xor(plaintext_bytes, key_guess)]]
````

This line represents the predicted Hamming Weight of the first-round AES intermediate value for a given key guess.

The solver then computes the correlation between this predicted leakage and the measured traces.

## 10. Attack Workflow

The practical workflow followed in this project is summarized below:

1. Verify the remote protocol using the provided interface.
2. Generate random 16-byte plaintexts.
3. Send each plaintext to the target through Option 1.
4. Decode and store the returned power traces.
5. Remove malformed or inconsistent traces.
6. Normalize the trace set for numerical stability.
7. Apply CPA independently to each of the 16 AES key bytes.
8. Reconstruct the full key.
9. Submit the key through Option 2.
10. Retrieve the flag.

## 12. Why the Attack Succeeds

The attack succeeds because the implementation leaks information through physical power consumption that depends on secret-dependent intermediate states.

AES itself is not vulnerable here. The vulnerability lies in the fact that:

* the plaintext is known,
* the first-round state depends directly on the key,
* the device leaks information correlated with that state,
* enough traces can be collected to distinguish the correct hypothesis statistically.

This is what makes CPA such a powerful technique in practice.

---

## 13. Security Implications

This challenge demonstrates an important lesson in embedded and hardware security:

> Strong cryptography is not sufficient if the implementation leaks information.

In real systems, protecting against side-channel attacks may require countermeasures such as:

* masking,
* hiding,
* random delays,
* power balancing,
* noise injection,
* hardware-level secure design.

Without such protections, even robust cryptographic algorithms can become vulnerable to practical key-recovery attacks.

## 14. Conclusion

This project presents a complete and realistic example of a **power side-channel attack** against an AES-128 embedded implementation. By combining chosen plaintexts, measured power traces, and statistical analysis, it is possible to recover the secret key without attacking the cryptographic primitive itself.

The challenge highlights the gap between **algorithmic security** and **implementation security**, and emphasizes the importance of considering physical leakage in the design of secure embedded systems.
